# OpenClaw 評分規則

## 評分架構

OpenClaw 使用「三道閘門 + 五維度加權」架構進行候選人匹配評分。

---

## 三道閘門（Gate Mechanism）

在計算五維度加權分數之前，候選人必須通過三道閘門。閘門限制分數上限或直接扣分。

### 閘門 A：相關性閘門

判斷候選人的專業領域是否與職缺核心領域相關。

| 情境 | 處理 |
|------|------|
| 核心領域高度相關 | 通過，不影響分數 |
| 跨領域但有關聯 | 通過，酌情扣 5-10 分 |
| **完全無關** | **最高分不得超過 25 分**，recommendation = "不推薦" |

判斷依據：
- 從職缺名稱和必要技能提取「核心領域」
- 對比候選人的工作經歷和技能列表

### 閘門 B：地理位置閘門（硬性條件）

| 候選人位置 | 處理 |
|-----------|------|
| 台灣且合理通勤 | 不扣分 |
| 台灣不同城市 | 扣 5 分 |
| 海外有台灣關聯（留學、曾在台工作） | 扣 15 分 |
| **海外無台灣關聯** | **最高分不得超過 40 分** |
| 地點未限定 | 跳過此閘門 |

### 閘門 C：資深度匹配閘門

| 情境 | 處理 |
|------|------|
| 職級相當（±1 級） | 不扣分 |
| **嚴重 Overqualified**（落差 >=2 級） | **扣 15-25 分** |
| 明顯 Underqualified | 扣 10-20 分 |

### 閘門疊加規則

三道閘門的扣分和上限可以疊加：
- 例：海外（max 40）+ overqualified（-20）= 最高 20 分

## 五維度加權評分

| 維度 | 權重 | 評分範圍 | 說明 |
|------|------|----------|------|
| 核心技能匹配 | 40% | 0-100 | 候選人是否具備必要技能？工作經歷中是否實際使用？ |
| JD 職責匹配度 | 25% | 0-100 | 過去工作職責是否與 JD 描述的日常工作相似？ |
| 公司 DNA 適配性 | 15% | 0-100 | 公司規模、產業、文化、技術棧匹配度 |
| 實際到職可行性 | 15% | 0-100 | 地理位置、職級適配、薪資預期 |
| 可觸達性 | 5% | 0-100 | LinkedIn+GitHub=100；LinkedIn only=60；都沒有=20 |

### 計算公式

```
raw_score = (技能 * 0.4) + (JD * 0.25) + (DNA * 0.15) + (可行性 * 0.15) + (觸達 * 0.05)
final_score = apply_gates(raw_score)  # 閘門限制
```

### 重要規則

1. **核心技能 + JD 職責（65%）得分低** → 總分不應超過 40 分
2. **資料不足處理**：
   - 職稱相關 + 公司對口 + 缺細節 → 45-55 分（觀望）
   - 職稱相關 + CPA/Big4 等標記 + 缺年資 → 50-60 分
   - 完全搜不到資料 → 15-20 分

## 公司 DNA 分析

在五維度的「公司 DNA 適配性」中，需要做以下四項子分析：

| 子維度 | 說明 | 示例 |
|--------|------|------|
| 公司規模匹配 | 候選人過去公司 vs 客戶公司規模 | 大廠→大廠 OK / 大廠→新創 注意 |
| 產業經驗匹配 | 候選人產業背景 vs 客戶產業 | 同產業 OK / 跨產業相關 注意 / 無關 差 |
| 文化風格推斷 | 從公司類型推斷工作風格 | 外商→外商 OK / 外商→傳產 差 |
| 技術棧一致性 | 從公司背景推斷技術環境 | 雲端 SaaS→雲端 SaaS OK |

## 職涯軌跡分析

在 AI 分析結論中必須包含職涯軌跡分析：

### 職稱晉升方向

| 類型 | 說明 |
|------|------|
| 上升型 | Engineer → Senior → Lead → Manager |
| 穩定型 | 同一級別持續深耕 |
| 橫移型 | 跨領域/跨職能平行移動 |
| 下降型 | 高階→低階（需特別標註原因） |

### 產業一致性

- 專注單一產業（金融、科技、醫療...）
- 跨界但有關聯（科技→金融科技）
- 頻繁跨界（可能缺乏深度）

### 任期穩定度

| 模式 | 說明 |
|------|------|
| 穩定 | 平均任期 >= 2 年 |
| 遞減 | 任期越來越短 |
| 不規則 | 長短交替，無明確模式 |

### 紅旗警示

- 頻繁跳槽（平均任期 < 1 年）
- 長期空窗（> 6 個月無解釋）
- 職級下降
- 從大公司到越來越小的公司

## 評等標準

| 等級 | 分數範圍 | Recommendation | 建議動作 |
|------|----------|----------------|----------|
| A | 85-100 | 強力推薦 | 立即聯繫，優先安排面試 |
| B | 70-84 | 推薦 | 安排初步聯繫，確認細節 |
| C | 55-69 | 觀望 | 有潛力但需更多資訊，可能適合其他職缺 |
| D | 0-54 | 不推薦 | 不符合當前職缺，但可建議其他適合職缺 |

## 三門後處理（Post-Processing）

AI 回應後需要進行三道後處理驗證：

### 後處理 1：名字黑名單

如果候選人名字為明顯的測試/假資料（如 "test", "xxx", "N/A"），直接設為 D 級。

### 後處理 2：分數合理性驗證

- score 必須在 0-100 範圍
- recommendation 必須是四個有效值之一
- 如果 AI 回傳的 score 與 recommendation 不一致，以 score 為準重新映射

### 後處理 3：Grade 映射一致性

確保 grade 和 score 對應正確：
```python
def score_to_grade(score):
    if score >= 85: return 'A'
    if score >= 70: return 'B'
    if score >= 55: return 'C'
    return 'D'
```

## 面談問題生成

AI 分析結果需包含 probing_questions，分三類各 2-3 題：

| 類別 | 目的 | 範例 |
|------|------|------|
| [初步] 條件確認 | 確認基本條件 | 目前在職狀態？期望薪資？最快到職時間？ |
| [技術] 深度確認 | 驗證技術能力 | 針對 JD 核心技能提問、針對缺失技能確認 |
| [文化] 適配確認 | 評估文化契合 | 根據公司畫像提問、團隊合作風格偏好 |
