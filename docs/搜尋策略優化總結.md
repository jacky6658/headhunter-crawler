# 爬蟲搜尋策略優化總結

## 一、搜尋流程

```
職缺關鍵字 → Brave API (site:linkedin.com/in/) → Google/Bing 備援 → GitHub 搜尋(技術職)
                         ↓
                    LinkedIn profile 爬取
                         ↓
                    Perplexity/Jina enrichment（補充經歷、學歷、穩定度）
                         ↓
                    關鍵字評分 + AI 評分 → 分級 (S/A/B/C/D)
                         ↓
                    去重（LinkedIn URL → Email → 姓名）
                         ↓
                    匯入 Step1ne 系統 → Telegram 回報
```

4 層搜尋備援順序：Brave API → Bing → Playwright (Google) → Google urllib

每層產出的是 LinkedIn profile URL，再逐一爬取個人資料做 enrichment 和評分。

## 二、職缺關鍵字搜尋策略

Brave API 產出 4 層 query（由精準到寬鬆）：

| 層 | Query 結構 | 範例 |
|---|-----------|------|
| 1 | `site:linkedin.com/in/` + job_title + location | `"Executive Assistant" ("Japan" OR "Tokyo")` |
| 2 | `site:linkedin.com/in/` + primary_skills OR + location | `("Executive Assistant" OR "Chinese" OR "Bilingual") ("Japan")` |
| 3 | `site:linkedin.com/in/` + primary_skills AND + location | 更嚴格的 AND 組合 |
| 4 | `site:linkedin.com/in/` + primary + secondary + location | 補充搜尋，加入次要技能 |

**關鍵字展開機制：**
- 每個 primary skill 從 `skills_synonyms.yaml` 展開最多 2 個同義詞
- Location 展開為城市變體（Japan → `"Japan" OR "Tokyo" OR "Osaka"`）
- Query 總長度上限 380 字元，超過即停止加入新 term

**Primary vs Secondary 分工：**
- **Primary（3-4 個）**：構成主搜尋 query 的 OR 條件 → 決定「搜到什麼人」
- **Secondary（5-8 個）**：只在第 4 層補充 → 擴大召回率

## 三、已知問題與優化解法

| # | 問題 | 解法 |
|---|------|------|
| 1 | 中文關鍵字搜不到外國人 | 關鍵字語言要匹配目標人選的 LinkedIn profile 語言 |
| 2 | 跨國職缺只搜一個地區 | 同職缺拆多任務：日本搜「會中文的人」、台灣搜「會日文的人」 |
| 3 | Primary 關鍵字太雜 | Primary 只放 3-4 個核心條件（職能 + 語言），Secondary 放擴展詞 |
| 4 | Brave API 422 | Query 上限 380 字元，同義詞限 2 個，OR terms 限 8 個 |
| 5 | GitHub 候選人全 D 級 | 提高加分上限、增加 stars/repos/followers 維度、放寬懲罰 |
| 6 | 非技術職搜 GitHub 浪費 | 行政/秘書類職缺跳過 GitHub，只搜 LinkedIn |
| 7 | 爬到 LinkedIn 登入頁垃圾資料 | 需加登入頁面偵測過濾 |
